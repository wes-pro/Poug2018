Resource usage of Oracle SuperCluster

Raport has been created on 2018-08-19.

Info Covers data from previous week and from databases running at the moment of data gathering.
More about gathering techniques

Data sources:
No. Source Example views Description
1 Dynamic views V$DATABASE, V$SGAINFO, V$PGASTAT, V$DATAFILE, itp. We use dynamic views to obtain basic database configuration parameters and current state. They are the only source of information about read-only databases (like DataGuard Physical Standby) for which AWR and Cloud Control do not get any data.
Each database is queried only once per week by script using local sqlplus client. If database is not started - information is not collected.
2 AWR repositories DBA_HIST_SYSMETRIC_SUMMARY, DBA_HIST_SGA, DBA_HIST_PGASTAT Historical performance data covers last week and comes mostly from DBA_HIST_* static AWR views. For example many metrics comes from DBA_HIST_SYSMETRIC_SUMMARY.
3 Enterprise Manager public views MGMT$METRIC_HOURLY Operating system, Exadata Storage Servers and some ASM metrics are obtained from Enterprise Manager public views (directly from database).
4 ASM dynamic views V$ASM_ALIAS, V$ASM_FILE, V$ASM_DISKGROUP To obtain full list of ASM files we use ASM dynamic views. These views are used to detect orphaned files, i.e. unused files littering ASM diskgroups.
Why just not use Enterprise Manager data for all statistics?

Many databases are not monitored by EMCC straightly after their migration to SuperCluster. There are test databases or databases migrated temporarily to prepare environment and procedures. These are some time not added to EMCC monitoring but they are important from overall system perspective because they do consume resources of SuperCluster.
Also read-only databases are not monitored by EMCC very well. For example Cloud Control does not gather memory utilization of these.
Last but not least, EMCC data is usually averaged while AWR data from V$SYSMETRIC provides better visibility of short-term phenomena.

CPU utilization by database

CPU load is shown separately for every domain and cpu pool. Databases are using 'single' or 'rac1n' cpu pools. 'Single' is used primary by single instance databases, vcpus from 'rac1n' are utilized by RAC One Node instances.

Individual charts depicts CPU load reported by databases running on particular domain and zone bound to one of two configured cpu pools. For visibility charts show only top consumers, rest is summarized under 'other' category.

Charts are stacked, i.e. performance metrics from every database using particular cpu pool are accumulated together. With some approximation this visualizes total CPU consumption in time.
Additionally, dotted line is added presenting CPU load obtained from operating system 'poolstat' command.

WARNING Accumulated (stacked) charts include only databases running at the moment data gathering scripts were running. This fact might artificially lower presented results because some databases will not be depicted here. On the other hand accumulation of data from different databases tend to overcalculate results because performance data is quantized to highest frequency before addition. Short-lived utilization spikes might not overlap in reality but they will on some of below charts.
For above reasons independent operating system metric is shown as dotted line allowing additional verification.
